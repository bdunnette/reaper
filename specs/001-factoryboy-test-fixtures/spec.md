# Feature Specification: Replace Static Test Fixtures with factory_boy

**Feature Branch**: `001-factoryboy-test-fixtures`

**Created**: 2026-06-11

**Status**: Draft

**Input**: User description: "replace static test fixtures with factory_boy"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Dynamic Mock Data for Autocomplete Tests (Priority: P1)

As a test engineer developing the `reaper` library, I want to use dynamic, customizable autocomplete responses in my test suites so that I can verify different search matches and location structures without manually constructing complex JSON dictionaries.

**Why this priority**: Replacing static dictionaries with factories makes autocomplete test setups simpler, cleaner, and less error-prone. It serves as the foundation for modernizing the test suite.

**Independent Test**: Can be verified by running `pytest tests/test_client.py` and checking that all autocomplete tests execute successfully using `factory_boy` models to generate input data, and assert on customizable fields.

**Acceptance Scenarios**:

1. **Given** a factory for autocomplete results, **When** I instantiate it with custom fields (e.g., `city="Seattle"`, `state_code="WA"`), **Then** it produces a structured dictionary matching the Realtor.com GraphQL format with the overridden attributes.
2. **Given** a factory for autocomplete results, **When** I instantiate it with no parameters, **Then** it generates valid default mock values (e.g., Austin, TX) matching the expected schema.

---

### User Story 2 - Dynamic Mock Data for Property Search and Detail Tests (Priority: P2)

As a library maintainer, I want to use dynamic property models and search result lists in tests so that I can easily test edge cases like missing description fields, various listing prices, different tax history arrays, and multi-page pagination.

**Why this priority**: Testing pagination and details involves many nested objects (e.g., address, description, tax history). Hand-coding static responses for multiple page loads or missing sub-properties is verbose and difficult to maintain.

**Independent Test**: Can be verified by running search and detail tests using `factory_boy` models and ensuring that tests cover both single property details and pagination edge cases successfully.

**Acceptance Scenarios**:

1. **Given** a search response factory, **When** generating mock responses for a list of size N, **Then** it generates exactly N distinct mock properties with sequential IDs or unique attributes.
2. **Given** a property detail factory, **When** a sub-property like `tax_history` or `description` is omitted/modified, **Then** the model validates the change correctly without breaking the API client tests.

---

### Edge Cases

- **Multiple concurrent calls (pagination limit)**: The search results factory must support generating sequential list elements when multiple pages are requested, ensuring each page's properties have unique identifiers and pagination ends cleanly.
- **Missing or optional attributes**: Factories must support creating sparse or partial responses (e.g., description being empty/null, tax history being an empty list) to test robust client parsing behavior.
- **Nested validation checks**: Nested factories (such as `AddressFactory` inside `LocationFactory` inside `PropertyFactory`) must correctly sync and output standard formats matching Pydantic expectations.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The test framework MUST define factories for all primary Reaper models: Autocomplete results, Property Search results, and Property Details.
- **FR-002**: Factories MUST allow callers to override any nested field or sub-structure during initialization (e.g., description, location, tax history).
- **FR-003**: Factories MUST produce output matching the JSON structure returned by Realtor.com's frontdoor GraphQL API to ensure compatibility with client parsing code.
- **FR-004**: The client test suite MUST replace all static dictionaries (`MOCK_AUTOCOMPLETE_RESPONSE`, `MOCK_SEARCH_RESPONSE`, `MOCK_DETAIL_RESPONSE`) with dynamic calls to the newly created factories.
- **FR-005**: All factories MUST be written using `factory_boy` and support standard attributes, SubFactories, and Lists where applicable.

### Key Entities

- **AutocompleteResultFactory**: Generates mock structures for location auto-completion, including attributes for city, state, area type, and slug ID.
- **PropertyFactory**: Generates detailed mock structures representing real estate properties, including nested address, description (beds, baths, sqft, year built, type), and listing status.
- **SearchResponseFactory**: Generates mock list responses containing a count, total, and a collection of properties generated by `PropertyFactory`.
- **PropertyDetailFactory**: Generates detailed property mock responses, including tax history list details.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Static mock dictionaries in `tests/test_client.py` are reduced from ~80 lines of hardcoded code to 0 lines.
- **SC-002**: 100% of existing client tests pass successfully when run under pytest with `factory_boy` generators.
- **SC-003**: Test coverage remains at 100% of existing lines, and test setup verbosity for custom response states is reduced by at least 50%.

## Assumptions

- pytest is the target test framework, and `factory_boy` will be added to development dependencies.
- The factories are only required for unit and integration testing; they will not leak into the production distribution package.
- The GraphQL client structure remains unchanged, only the mock data providers are replaced.
